Online Version Differences

This project was originally developed as a collaborative exercise, with teams doing a whole project, using class time as a meeting space for people to split up work responsibilities.

This form of collaboration requires meetings at least three times a week, and that simply isn’t compatible with the scheduling flexibility students depend on in an online course.

In the adapted format, I provide roughly the amount setup work that might be done by other members of a team. For writing queries, I let you choose those that you are best prepared to write or are most interested in, which is very much like a project “signup” in some Agile processes. You will be working in Git, and sharing your work by committing and pushing it to a common repository. I expect your work to be self-contained and to work in my copy of the repository just as it does in yours. These mechanics are similar to what they would be on a team.

As you go through this online adaptation, I encourage you to think about how you would have come up with anything I provide: you or someone like you would be expected to create these artifacts when building a product from the beginning.

The good part about a group project is that no one should be overwhelmed by the work: someone will be there to support you. I will perform that role. Let me know how you’re doing!

Something I like about this adapted format for the project is that everyone has the chance to see the richness of SQL: instead of one member grabbing the hardest or easiest or most-fun, you can all see those parts. I think this can be a very good experience.

Support

The goal of the project is to let you experience how all the technologies and techniques you have practiced in this class–from requirements gathering to answering questions with SQL–fit together.

This is an introduction that covers a lot of material. I hope that as you go over it, each step will feel familiar. At the same time, I hope this gives you new perspectives and raises new questions as you connect the pieces from the course.

I am here to answer any of these questions. Send an email or arrange a Zoom. The online format makes it hard for me to know what areas you are confident about and what areas I have not explained in a way that resonates for you. A short email or a Zoom can help us quickly bridge this online gap, and I hope would be rewarding for both of us.

Let me help.

Project Mechanics

GitHub

Throughout this project, you will be working in a single GitHub repository. Your work will build on earlier commits.

I will create a repository specifically for you, and invite you to collaborate on it. This means we can both push to the same branch, which is how members often work within a team. I think it is fine if you stick with the “main” branch, but I can work with other branches if you let know how you want me to use them.

Codespace and PostgreSQL

The repository will be DevContainer/Codespaces enabled, so it will configure a PostgreSQL database server for the project to work with. This will start when you start your Codespace.

The database will start empty; it is your responsibility to update the SQL/scripting so it populates successfully from scratch.

Supporting Files

I will provide:

  • reset.sh (to drop and recreate the database)
  • a script to run both the SQL to create tables and to populate tables (create.sh)
  • some SQL for creating and populating the database so it matches the sample data

You will have to modify or add to these to create a running sample of the database for the customer to evaluate.

Assignments

The assignments are provided in Canvas. They start with some design questions, then move into implementation.

For each assignment, you should commit and push/sync your code in GitHub, then indicate you are complete in Canvas.

Artifacts

Remember that you are building code (mostly SQL) that a customer would try out to evaluate your prototype, and that would be used in the future to work with the database after it has grown to have much more data. The queries are not questions that need an answer today, but tools that will provide an answer today and again in the future.

I will evaluate your code in my own Codespace. I can see only what you have published, and evaluate only what happens when I run your scripts. The Codespaces environment makes it much easier to bundle your work and show it to someone else, so you should not have challenges with this, but I may ask you to look into any issues that show up when your code is run in a different Codespace.

Sync

I will be looking at our shared repository on GitHub. Your Codespace creates a clone of that respository–the one we share is the remote “origin” in the Codespace.

You must “Sync” in VSCode (or “git push” at the Terminal) to copy your work up to the shared repository.

If you want to confirm what we can both see, visit the GitHub repository. (This is the page that you start the Codespace from.) That page should show the synched versions of files and the log message from your latest commit.

Notes and Special Care

Some areas of this scenario require special care:

  • people appear in lots of reports–since these are often the same people, normalize (or look for a single source of truth)
  • a Sale has a many-to-many relationship with Merchandise (one order can be for more than one item, and different customers may order Merchandise at different times on different orders)
  • be careful that the “Quantity” attribute is attached to the appropriate entity

Project

Overview

The project is to construct a database for a sample organization working on climate action. The process of developing the database will follow the model presented in class: you will suggest requirements on the basis of an initial interview; then you create a data model that will go on to become the structure of the database. You will write SQL to create, populate, update, and report on the database to address the customer’s needs.

Artifacts

I will invite you to collaborate on a shared repository.

This repository will be the Single Source of Truth for your project. Git keeps a history of your commits, so I encourage you to commit anything you think is interesting (so it is logged). Git’s history includes deleted files, so feel free to delete any files or parts of files that are distracting or that you don’t want to show as your final product.

Here are the artifacts I will be asking for in the project. The order follows the Analysis/Design/Implementation methodology. You should commit all artifacts to your repository and sync with the origin on GitHub.

Requirements Statement

A Statement of Scope is not required. If you write one, I will provide feedback on it. If you want to try seeing how a generative AI produces a Statement of Scope, you are welcome to experiment with that, too. If you use generative AI, watch that the AI does not try to commit you to features or standards that the customer did not ask for. (These features would probably raise their costs without providing value.)

Design

You will adapt the design in a couple of places. The project includes a Diagrams.net diagram of the sample documents (without normalization, relations, etc.).

If you use the design tools to help with your table additions, please add/commit your design files so they can be part of your project documentation!

SQL

The project will require several batches of SQL; each will have a different responsibility and will be needed at different stages of the database’s lifetime. The project framework includes these files; you should adapt them as necessary.

  • SQL statements to complete the table structure from an empty database
  • SQL to populate the database with the sample data
  • queries to answer questions the organization is interested in

Tools

Analysis and Design

During Analysis and Design, you will be working with the sample data (in .html) and the ERD (in .drawio) format. The DevContainer includes the hediet.vscode-drawio in your Codespace so you can work with .drawio files directly. However, VSCode may not be the best tool for this work: for the ERD you may want more screen space than the Codespace window.

The Codespace treats .html as source; the samples may be easier to read in preview or in a browser directly.

You can download both the .html and .drawio files from the GitHub repository viewer (not the Codespace) to your local machine and work with them there.

Or–even better: clone the repository locally and work in the cloned directory. You can use VSCode or the Git command line to sync.

SQL

Use the Codespace and the web-based VSCode when writing SQL: it has the PostgreSQL installed and set up, and SQLTools configured to work with it. I will use this Codespace configuration to test your code.
